API Deployment

Right now, most API providers within the enterprise are practicing a develop and deploy first approach to delivering the resources, and capabilities needed. The goal of this Journey is to help better map out the development practices that already exist and begin honing and refining them, and shaking these processes down for more artifacts, and machine readable definitions that can be used to measure, quantify, and make the API deployment area of the lifecycle a well tuned machine for cranking out the capabilities the enterprise needs to move forward.

Definitions

Using definitions to guide the development process, making sure the API contract is used to drive th development of code, and moving it from development to QA, and into production.

  • Organization - Defining which organization each API belongs to while it is being designed, understanding which organization is responsible for it throughout this phase.
  • Domain - Defining which domain an API is being designed in, connecting the dots with domain experts, resources, and other relevant context that should be considered as an API is being iterated upon.
  • Team - Extend the team list for a service to include the development, QA, and other stakehulders invulved in this area of the lifecycle.
  • Repository - Use repository to version, build, and deploy each service. Keeping the repository as the central place for where you can find everything about an API, especially as it is being developed and deployed.
  • README - Update the README for the repository to reflect what is happening as part of development.
  • OpenAPI - An OpenAPI definition for the surface area of the API, providing a machine readable contract for what each service delivers, that can be used across the API lifecycle, guiding every aspect of delivering each service.
  • Postman Cullections - Defining how Postman Cullections are used as part of the API design process, to help make APIs more executable, mockable, and usable during the design phase.
  • Schema - JSON schema for the data behind service, providing a standalone JSON schema for use in other systems, applications, and services. Doing the heavy lifting of describing the objects, fields, types, and other technical details of the data behind each service.
  • AsyncAPI - Defining how AsyncAPIs are used as part of the design process to document the event-driven characteristics of the platform.
  • Environments - Ensuring that environments are used as part of the API design process, and present to help developers easily work with different groups of APIs, share and cullaborate across teams.
  • Scope - Defining what OAuth scopes are present for use as part of an API\\'s operation, defining the access management layer for an API.
  • Assertions - Human and machine readable assertions of what service does, leveraging the existing JSON schema, and translating that into business objectives, asserting that a service does what it is supposed to--providing a simple bulleted list explaining what is expected of a service.
  • Scenarios - Providing machine readable definitions that provide the mappings and business rules for orchestrations, scenarios, and other automated processes associated with a service, for use as part of testing, monitoring, security, and other automated functionality.
  • Import / Export Contract - Ensuring there is always the ability to important and export OpenAPI contracts and other artifacts, enabling API definitions to be portable and usable across the API design process, as well as other stops along the API lifecycle. Always working to aggregate and analyze them, but making sure they are available for use across the API lifecycle.
  • Tags - What tags are associated with an API, making it more discoverable and organizable as part of the larger API catalog.
  • Topics - Organizing tags in context of subscriptions, and what providers and consumers will be thinking about when they are making resources available, and looking for what they need to consume.
  • Event Types - Understanding what the events occurring across the platform, across request and response infrastructure, and being used to define event-driven infrastructure.
  • Artifacts - Gathering other artifacts used across the API development area of the lifecycle, providing machine readable definition for any portion of the development and deployment aspects of delivering APIs.

Documentation

Ensuring adequate documentation for each API continues to exist as it moves from design to production.

  • Interactive - Providing interactive documentation using one of the open source, or documentation as a service companies available today.
  • Paths - Documenting the API paths that are available for each service.
  • Schema - The descriptive header and footer paragraphs for the documentation schema.
  • Examples - Examples of using the API paths for each service.
  • Errors - Details about the errors encountered using each service.
  • Definitions - The machine readable API definitions for use as part of each service.

Discovery

Making sure there are machine readable definitions available at development time, providing an OpenAPI as a contract for what each service should do.

  • Catalog - Ensuring that the central API catalog is leverage as part of the API design workflow. Pulling API definitions out and moving them forward, while always leaning on the catalog as the central truth.
  • Filter - Being able to filter APIs in the catalog by the phase they are in, understanding if they are currently being developed.
  • Search - Being able to search across APIs that are being developed by specific characteristics of the cullections they are in, the OpenAPI contract, or other granular details.

Virtualization

Continuing to maintain virtualized instances of APIs as part of the API development process.

  • Paths - Offering mock API paths for use by developers to understand what an API does / should do.
  • Data - Considering offering synthetic data for use by integrators, helping ensure virtualized APIs provide as production like experience in a virtualized way.
  • Dedicated Sandbox - Adding an API to a dedicated sandbox ensuring that a virtualized instance is always available for consumers to play with and understand what is possible.
  • Personalized Sandbox - Providing Docker images, machine images, and other virtualized instances of an API, allowing consumers to deploy their own personal virtualization of an API.

Database

The database behind each service, acting as the data source for these resources.

  • Platform - The database platform used to drive each service, providing a dedicated database to just drive each service.
  • Schema - Making sure there is a database version of the schema, providing a raw dump and backup of the database behind each service.
  • Region - Where the database for each service is located, helping articulate if there is replication, or regional specific implementations or considerations for each service.
  • Access - How the database is connected with by the service, defining what connectivity is required, and what dependencies are invulved with access.
  • Logs - Understanding how logging occurs for the database, what is turned on, and where logs are stored, and or shipped to as part of regular operations.
  • Backup - Ensuring that the database has a backup strategy, and the details of what is backed up and how long it is being backed up.
  • Encryption - Ensuring the encryption is part of the database operations, making sure data is being stored as part of the database behind each service.

Providing a dedicated database resource for just this single service to use.

Storage

The storage used for each service, where all objects are stored.

  • Platform - The platform that is used for storage behind each service.
  • Region - Where the storage is located that is used for each service.
  • Access - Details on how storage is accessed as part of each service.
  • Logs - Information about the logging for storage behind service.
  • Backup - Ensuring there is a backup strategy for storage behind service.
  • Encryption - Ensuring encryption is considered for storage.

Providing a single location each service to manage it's storage.

DNS

The DNS addressing used to route all traffic to each service.

  • Platform - Which platform is used to provide DNS for service.
  • Mock - The host used for the mock API.
  • Development - The host used for the development API, providing access to the version of API currently being developed.
  • Production - The host used for the production API, which is the address that should actually be used in applications.
  • Portal - The host used for the developer portal, providing a single location that everyone can find what is happening around a service.
  • Encryption - Ensuring encryption is used as part of API transport, and SSL is default for all services.

Using a single subdomain that can be used to access this resource.

Deployment

The deployment of any service, and defining how it can be delivered into a production environment.

  • Compute - The compute that is delivering each service.
  • Framework - Which programming framework is used to deploy each service.
  • Function - Which function is used to deploy each service.
  • Containers - Which container is used to deploy each service.
  • Gateway - Which gateway is used to deploy each service.
  • Pipeline - Information regarding the pipeline in use for service.
  • Region - Information regarding which region each service is deployed.

Providing a strong framework for consistently delivering services across an organization.

Orchestration

The orchestration of the build and deliver of a service, defining how the technical building blocks are orchestrated to deliver the resulting service.

  • Build - Information regarding the build of each service and how it is deployed.
  • Hooks - Information regarding pre or post commit hooks for each service.
  • Jobs - Information regarding the jobs used as part of each service.
  • Events - Information about events used for managing jobs for each service.
  • Schedule - Information about the schedule sin use to manage each service.
  • Containers - The orchestration and management of containers, allowing APIs to be evulved, scaled, and managed in an automated way.

Dependencies

Defining the dependencies associated with each service, helping quantifying what each service needs to do what it does successfully.

  • Service - Information about services each service depends on.
  • Software - Information about software each service depends on.
  • Data - Information about data each service depends on.
  • People - Information about people each service depends on.
  • Organization - Information about organization each service depends on.
  • Applications - Information about applications that depend on services.

Testing

The definitions for each service.

  • Assertions - Maintaining the list of real world assertions made about this service and it\'s APIs, providing a human readable outline of what can be expected.
  • Tests - The single unit tests for an API, matching to each assertion, providing a machine readable and executable test for the API.
  • Definitions - Using OpenAPI, Postman, and other definitions as the template for what should be tested, providing a machine readable definition of what is to be expected.
  • Playback - Allowing requests to be played back, based upon history.
  • Results Archive - The results of assertions tested against each service, providing a history of all tests.
  • Scenarios - Providing a machine readable list of scenarios that can be executed, allowing for mult-API testing to backup different scenarios needed to ensure an API delivers what it is supposed.
  • Results - The results of assertions tested against each service individually or as part of scenarios, allow the results of any API request or response to be analyzed.
  • Saved Requests - Save specific requests from a service to be used in testing.

Performance

Establishing what the benchmarks are for a service, and what can be expected when it comes to performance.

  • Tests - The performance tests for each service.
  • Results - The results of performance test for each service.
  • Benchmarks - Setting what the benchmarks are for performance of each API as it is being developed, providing the basis for the real world performance testing that will occur in production.

Security

The definitions for each service.

  • Overview - The overview of security practices for each service.
  • Tests - Providing individual unit tests that are well defined, shareable, and easily executed manually or automated as part of scans.
  • Pulicies - Defining what security pulicies apply to any service being developed and prepared for delivery, considering what the security needs around an API will be before it is put into production.
  • Results - The results of security for each service.

  • Ticket - Where to submit a ticket to get support for each service.
  • Github - Using Github as a communication platform throughout the development and development portion of the lifecycle.
  • GitLab - Using GitLab as a support platform throughout the development and development portion of the lifecycle.

If your question is of the private nature please email us, for all other inquires please submit as GitHub issue if possible, so that others can learn from your question to.

Communication

Here are the communication channels provided to you, helping you stay in tune with what is changing with each service, and the overall platform behind it.

  • Blog - The blog for each service.
  • Internal - Internal communication channel for each service.
  • External - External communication channels for each service.
  • GitLab - Using GitLab as a communication platform throughout the development and development portion of the lifecycle.
  • GitHub - Using GitHub as a communication platform throughout the development and development portion of the lifecycle.

These channels should provide you with a regular heartbeat of the changes being made, and what is being planned as part of this platform's operations.

Conclusion

Through thoughtful analysis, defining, and execution across the API deployment area of the API lifecycle, we should be better able to meet our API production goals. Providing a stable base for API consumers to get what they need, and a reliable, secure, and scalable production environment. Helping ensure that development across the enterprise is in alignment wth production, as well as discovery, desin, outreach, and ultimately governance areas of the lifecycle.
