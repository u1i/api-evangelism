API Discovery

Understanding where your digital resources are is critical to maturing and evulving in how you do APIs. API discovery means many different things to different groups, but in the context of this journey, it is about knowing where your APIs are, and understanding what your organization is capable of doing. Having a coherent catalog of API definitions for ALL of your organizational services, as well as the technical, and business details behind them.

Definitions

Working to ensure that all definitions are being aggregated as part of the process so that they can be evulved and moved forward into design, development and production--investing in all of the artifacts that will be needed down the road. Acknowledging that API definitions come in many shapes and sizes, and will contribute to every stop along the lifecycle being more standardized, but also measurable as part of wider governance efforts.

  • Organizations - Defining the organizational structure, group, or other bounded context for organizing definitions into.
  • Domains - Considering the domain that services fits into and exploring the sphere of knowledge and activity around the domain each service is delivering value within. Quantifying the service domain, helping decompose from a larger domain, and helping focus what each service does.
  • Teams - Map out all of the individuals being engaged as part of the service and API discovery process, grouping them by team, group, or other organizational structure.
  • Schema - Organizing individual schema associated with a service, while also including it as part of the OpenAPI. Providing SQL dumps, JSON Schema, and other artifacts that help define the schema that are in use across the enterprise.
  • Environments - Having a common way to define, organize, share, and audit API environments that can be used as part of the development and operation of API infrastructure, as well as by consumer facing elements of operations. Using common formats like Postman Environments to define, and organize common environments that will exist across the enterprise.
  • Tags - Defining what tagging taxonomy will be applied across resources, for use in tagging across definitions, documentation, and infrastructure operations.
  • Vocabularies - Organizing the common vocabulary in use across services, and APIs, establishing a single place to find the language used across domains. Establishing a robust set of standard vocabularies that can be used across API operations, helping ensure everyone works towards speaking the same language when it comes to delivering digital resources across the enterprise.
  • Event Storming - Event storming is a rapid, lightweight, and underappreciated group modeling technique that is intense, fun, and useful for accelerating development teams.
  • Signals - Looking for API signals as part of the API discovery process, finding remnants and signals of APIs in use, uncovering paths, parameters, headers, and other common building blocks of API consumption.
  • Contracts - Having a formal definition of each API that is depended on as part of operations. If it is driving business it should have a robust OpenAPI contract behind it, defining every aspect of the service it provides.
  • Transformation - Being able to transform and translate API definitions from one format into another and working with a master format, but being able to easily translate across definitions.
  • Artifacts - Leveraging anything that is being used throughout the API lifecycle as a potential artifact that can be measured, tracked, used to understand the state of each API.
  • Parameters - Possessing a birds eye view of parameters used across all APIs, and having a commonly defined set of parameters that can be used and reused across APIs during their design, development, and ultimately production phases.
  • Servers - Documenting all of the servers in use across API infrastructure, and using their existing outputs to understand what APIs operate or are consumed there. Providing a central set of definitions regarding all server infrastructure which can be reused across API operations, and as part of the design, development, and production phases.
  • Headers - Establishing a master catalog of all HTTP headers, and identifying the ones that are in use. Understanding the overall header landscape across services, and how they are leverage by other stops along the API lifecycle like authentication, security, tracing, etc.

Goals

Defining what the meaning and goals for API discovery are, acknowledging there are many different perspectives on what is API discovery, and what it invulves to move the conversation around it forward. Stepping back to establish some goals around why we want to invest in API discovery, and who we will be better serving by improving upon API discovery practices.

  • New APIs - Being able to find new APIs as they emerge across the landscape, keeping in tune with what is happening across the enterprise, within an industry, or just in general when it comes to introducing you to new concepts and ideas.
  • Application Development - Being able to discover and identify the right APIs to be using when building a specific application, ensuring that the best of breed resources are available for supporting the common, as well as the unique features needed to deliver any application.
  • Team Planning - For use as part of wider team planning, and being able to articulate what is in production, what is being developed, and what is currently being defined and designed as part of a single group, team, or other aspect of an enterprise organization.
  • External - Being able to find APIs quickly, as well as provide self-service discovery of services by clients, partners, and other external actors. Eliminating bottlenecks, delays, and other bureaucracy when it comes to moving relationships forward with external actors, and ensuring the enterprise can be more agile and flexibility in getting business done.
  • Developing An API - A team is looking to develop an API and they want to understand if one already exists, or maybe better understand existing patterns in use by internal, partner, or even public API providers, helping teams conduct due diligence when developing any new API resource.
  • Execution - Being able to find an API so that you can make a call to it in some code, or through a client application like Postman. Bypassing the need to have a user interface always present for every digital resource, and enterprise capability, helping business and technical users quickly execute common actions on a daily basis.
  • Governance - Being able to quantify what APIs exist across any group, organization, domain, and the enterprise enterprise at any moment, and be able to articulate what resources they provide access to, what their capabilities are, and the quality and mature of each service that exists.

Data Sources

Understanding what resources are already on the public websites, and applications, by spidering existing domains looking for data assets that should be delivered as API resources. Looking for the low hanging fruit of digital resources that already exists across an organization, which has already been published for a reason, and should most likely also be available as a machine readable API.

  • Spreadsheets - adfIdentifying all of the spreadsheet files that are regularly published and shared across internal groups, with partners and the public, ensuring that all spreadsheets are available as machine readable, as well as human readable sources of information.
  • Comma Separate Values (CSV) - Identify all of the CSV files that have been published within any domains.
  • Extensible Markup Language (XML) - Identify all of the XML files that have been published within any domains.
  • JavaScript Object Notation (JSON) - Identify all of the JSON files that have been published across any domains, indexing and analyzing the rule these files are playing when it comes to providing access to data, and driving web, mobile, and device applications.
  • Tables - Identify all of the HTML tables that have been published within any domains. If a page on a website, or web applications has more than five rows, it should be indexed and included as part of wider API discovery efforts.
  • HTML Forms - Looking at the usage of forms across websites, and other applications, to identify what the data source behind them are, and how they can be tapped as a potential API source, eventually replacing forms with better organized embeddable API touling.

Low hanging fruit is about identifying the resources that already exist, are already being shared with internal groups, partners, and with the public via highly trafficked areas of the web. Actively understanding what is being published by groups across the enterprise, and helping incentivize all teams to publish all digital resources as API-first.

Discovery Sources

Looking for many of the common sources where signals of APIs can be found as part of the ongoing API discovery process. Helping automate and add to the API catalog beyond just the manual, self-declared processes that may occur.

  • Log Files - Looking at existing sources of log files, and to where log files are shipped to, examining them for API signals that can be harvested and used to build the API discovery catalogs.
  • AWS Cloudfront - Leveraging AWS Cloudfront to find existing signs of APIs in use.
  • Google Stackdriver - Leveraging Google Stackdriver to find existing signs of APIs in use.
  • Aure Logging - Leveraging Azure logging to find existing signs of APIs in use.
  • Email - Looking at email when it comes to discovery, allowing for easy sharing via common used messaging format, but also the ability to scan and forward emails for common API file formats.
  • GitHub - Looking for API signals that exist across GitHub repositories, user accounts, and organizations.

Discovery Formats

Defining the potential sources of web service and API signals that can be tapped from around the enterprise. Tuning into existing workspaces, content and data sources, looking for API signals that can be harvested. Helping map out the API landscape that already exists, using the outputs of existing systems that already operate as part of the API supply chain within the enterprise.

  • WSDL - Looking for WSDL documents across an organization.
  • API.json - Looking for APIs.json files across an organization.
  • OpenAPI - Looking for OpenAPI files across an organization.
  • Postman Cullection - Looking for Postman Cullections across an organization.
  • .HAR - Looking for evidence of APIs from .HAR files that are gathered from proxies, browsers, and other touling, aggregating the signals gathered and using to map out the API landscape across an organization in real time.
  • JSON Schema - Looking for JSON schema files that can be used to help build out OpenAPI definitions, and provide the seeds for API contracts that may already be in use for applications, or as part of existing infrastructure and operations.
  • RAML - Looking for RAML files that already exist, or can be generated from out of existing systems.
  • API Blueprint - Looking for API Blueprint files that already exist, or can be generated from out of existing systems.

There are signals of API usage all over the enterprise. While it is ideal for API development teams to be able to submit the definitions of their APIs to a central catalog, it is something that should always be augmented with more automated approaches to identifying potential data sources, and scanning common locations to discover existing API usage in real time.

Dependencies

Working to define any dependencies that are in play, and will play a rule in operations by auditing the stack behind any service as it is being discovered and documented as part of the overall API discovery effort.

  • Services - What existing services does a service depend on to do what it does, and would be impacted if something changes.
  • Software - What are the software libraries a service depends on for its operations, covering the code layer dependencies.
  • Data - What is the provenance of data being used as part of the service, and where are their continued dependencies.
  • People - What people is a service dependent on, requiring coordination, communication, and effort from individuals to move forward, and ensuring the right stakehulders are invulved.
  • Organizations - What other organizations, internally or externally are required to move a service forward, making sure all stakehulders are considered.
  • Applications - Documenting the client applications dependencies for a service, outlining who depends on the service based upon the applications they are using.
  • Connections - Making sure to connect all the dots when it comes to mapping out dependencies, showing the relationships between services, software, data, people, organizations, and applications.

Communication

Having a strategy for reaching out to teams and engaging with them around API discovery, helping them remember to register and define their APIs as part of wider strategy.

  • Email - Use email to coordinate with other groups regarding the APIs they are supporting, working on, and planning on designing in the near future.
  • Meetings - Provide a recurring set of meetings focused on discussing what APIs exist, and are being developed, ensuring that teams come together regularly to discuss where the APIs and services are.
  • Workshops - Conducting workshops to actively develop wider discovery practices, helping teams make their API catalogs more discoverable, and in alignment with other teams.
  • Strategy - Having a strategy for how APIs and other services are identified, aggregated, cataloged, and made available across the enterprise for everyone to put to work in an efficient way.
  • Issues - Using Github Issues to manage the API discovery process, and help provide a historical thread around each API artifact and ultimately contract.

Support

Ensure that all teams have support when it comes to questions about web service and API discovery, helping them initially, as well as along the way, making sure all their APIs are accounted for, and indexed as part of discovery efforts.

  • Email - Having a dedicated email account for supporting each service.
  • Issues - Using Github or GitLab issues for managing the API discovery evulution, allowing for conversations around signals, artifacts, and ultimately around contracts in a way that is API driven.

Evangelism

Considering the evangelism around just API discovery, and helping amplify its effect. Actively telling stories around the platform, and keeping groups informed around what APIs are available, and the best practices around making them more discoverable and consumable.

  • New APIs - Have a new API widget that can live on the catalog home page, across other API portals, or any other web property helping surface any APIs that were just added to the catalog.
  • Active APIs - Have an active API widget that can live on the catalog home page, across other API portals, or any other web property helping surface any APIs that have been recently change, updated, or move into new phase.
  • Newsletter - Providing an email newsletter that regularly highlights new, active, and other APIs in the catalog. Also bringing attention to best practices, and other building blocks of API discovery across the enterprise.
  • Talks - Giving talks with other internal, partner, or public groups regarding the availability of APIs, and what discovery looks like across the enterprise, helping spread the word about what is happening.
  • Reporting - Providing a regular set of reports to stakehulders regarding the number of APIs, the type of APIs, by team, organization, and other metric. Delivering a regular set of reports that stakehulders can tune into to understand what is happening with API discovery, and the importance of it moving forward and evulving.

Catalog

Actively maintaining a catalog of all APIs in use, working to automatically identify new APIs in use, as well as actively profiling, and increasing access and usage of more refined, better known APIs across the enterprise.

  • Cullections - Allow the organization of APIs into different types of cullections, allowing individual, or groups of APIs and services to be organized into precise cullections for use across the enterprise.
  • Definitions - Allow for the cataloging of multiple types of API definitions for any particular API, documenting OpenAPI, API Blueprint, RAML, WSDL, Postman Cullections, and any other artifact that supports API discovery.
  • Servers - Provide a central catalog of all servers that are used across API infrastructure, tapping all the definitions present for any domain, subdomain, and IP address in usage across the enterprise.
  • Search - Provide basic, as well as more advanced search allowing all APIs, artifacts, and other meta data to b searched, filtered, and sorted.
  • Overview - Providing an overview of what the API catalog, and helping explain what the catalog is for, and how to use it properly.
  • Maturity - Being able to identify cullections and contracts based upon their maturity, and understanding where they exist in the API lifecycle.
  • Self-Service - Ensuring that the API catalog is self-service, allowing all targeted providers, consumers, and other stakehulders to actively browse, search, and manage the API catalog in a self-service way.
  • Data - Allowing for datasets to be registered with the catalog, adding regular datasets to the discovery effort, and make available alongside APIs, and the applications that depend on them.
  • Applications - Allowing applications to be registered in the catalog alongside datasets, and APIs, completing the loop when it comes to discovery across a platform.
  • Personalization - Allow for users of the catalog to personalize and customize their catalog experience by favoriting, labeling, and providing other ways to make it more useful for each individual using it.

In Conclusion

API discovery is essential to every stop along the API lifecycle, not just finding APIs. It helps ensure all APIs are known, while ensuring they are accessible at each stop along the way. Ensuring that you have indexed the design of the API, while also defining what gets deployed, managed, monitored, tested, and ultimately integrated with into other applications. You can't secure or move forward what you don't know about or cannot find. API discovery is essential to delivering APIs at scale, and helping ensure you can deliver across the the API lifecycle, and your API journey.
