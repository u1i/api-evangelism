API Governance

API governance is the ultimate goal of wanting to establish a formal strategy for delivering APIs consistently at enterprise scale. It is what is needed to quantify, measure, and ultimate steer the enterprise API ship, and guide teams forward when it comes to delivering APIs at scale. Governance will mean many different things to different organizations, and team members, but there will always be some essential building blocks we should be looking at to understand where we are succeeding or failing with our API efforts.

Structure

Working to understand the scope and structure for governance operations, establish the right approach to understanding API operations across an organization, and be effective when working with many different teams, operating at different levels of maturity.

  • Resources - Defining the resources that are available to realize governance. Being realistic about what is possible, and what it will take to actually make change across the organization.
  • Distribution - How distributed or centralized will API governance be. Reflecting the organization(s) it represents, but appropriate for moving forward the conversation as desired.
  • Team - Making sure there is someone present from every team, representing the governance effort, requiring that all teams have a seat at the table, providing feedback on the overall governance, but also acting as coach and leader within each team when it comes to governance.

Discovery

Ensuring that all APis are discoverable and able to be measured as part of overall governance. If it isn\'t known about it can\'t be monitored, tested, secured, and analyzed for the value it generates across an organization.

  • Topics - Organization resources into meaningful tagged groups based upon logical groupings, and crafted for consumption via synchronous and asynchronous subscriptions, using to define the most relevant buckets of information.
  • Organization - Have a clear map, as well as understanding of the organization, and the breakdown of API stakehulders across those teams. Possessing a machine readable way that is integrated with existing directory sulutions, as well as overlays and takes advantage of existing services and software in use.
  • Domain - Having a clear definition of the domains invulved with enterprise operations. This is a tricky world to define because it is spread across Internet domain, as well as expertise domains, which may or may not overlap. Regardless, working to have a clear map of domains across the enterprise, and an understanding of how API infrastructure is operated across them is essential.
  • Team - Having a clear breakdown of teams across organizations, who are building with a variety of domains is important to being able to govern API operations. Understanding which teams are responsible for which APIs, and how team breakdown positively or negatively impacts the evulution of API delivery across the enterprise.
  • Repositories - Documenting, spidering, and analyzing repository usage across all projects, within all team and organizational structures. Using the repository as the small unit of compute when it comes to what is happening with each service, and leveraging code management practices to keep all the moving parts of the enterprise working together.
  • API.json - Leveraging the machine readable API discovery format to organize and index APIs, providing a format for quantifying, measuring, and ultimately guiding APIs, and teams across the API lifecycle. Using APIs.json as the wrapper for organizing all other human and machine readable artifacts gathered across the lifecycle.
  • OpenAPI - Ensuring that OpenAPI contracts are available for all services being operated across the enterprise, acknowledging they will be at different levels of maturity, but always pushing them to be refined, evulved, and iterated upon at all stops along the API lifecycle so that they can be measured and used as part of API governance.
  • Schema - Leveraging a central repository of JSON schema for use to define the underlying data layer, but also leverage for defining and understanding the rules needed to be applied across the API lifecycle.
  • Postman Cullections - Using Postman Cullections definitions as an executable representation of an API capabilities. Postman Cullections do not offer much in the way of discoverability, but do provide a machine readable artifact that can represent the usability, maturity, and value of an API.
  • AsyncAPI - Using ASyncAPI as a way of understanding the surface area of event-driven APIs, and using it to define operations, measure the evulution of each API, and how it fits in with the overall governance strategy.
  • Environments - Tracking machine readable environment artifacts from the touling and services used across the lifecycle. Establishing a common and portable format for describing the environments needed across all areas of the API lifecycle.
  • Scopes - Defining what OAuth scopes are present for use as part of API operations, defining the access management layer for an API. Providing the benchmark of data access and filtering available across the enterprise and groups.
  • Assertions - Human and machine readable assertions of what service does, leveraging the existing JSON schema, and translating that into business objectives, asserting that a service does what it is supposed to--providing a simple bulleted list explaining what is expected of a service.
  • Scenarios - Providing machine readable definitions that provide the mappings and business rules for orchestrations, scenarios, and other automated processes associated with a service, for use as part of testing, monitoring, security, and other automated functionality.
  • Tags - Aggregating, organizing, and evulving tags that are used across all stops along the API lifecycle, and making them more meaningful and usable throughout the entire stack, so that they can be used to searching, organized, and make sense of operations at scale.
  • Event Types - Taking all properly labeled and organized resources, and extract the types of events that are possible with the design of each API, and develop, maintain, and evulve a list of all event types that can occur.
  • Artifacts - Gathering other artifacts used across the API development area of the lifecycle, providing machine readable definition for any portion of the production side of delivering APIs reliably.
  • Guidance - Providing ongoing guidance when it comes to what definitions should be used, produced, aggregated, shipped, and used to help steer the ship as part of API platform governance. Pushing the known API lifecycle to produce relevant artifacts that can be measured.
  • Results - Producing quantifiable and reportable results regarding which teams are using definitions, and how they are being applied in alignment, or out of alignment with wider API governance guidance.

Design

Work to provide guidance regarding API design, and consider how to measure if it is being implemented by teams. Working from group level, or centralized API guidance, and working to refine regularly across teams as things change.

  • Parameters - Having a common approach to defining parameters, and organizing them as a central catalog of schema that can be reused, and quantified as part of enterprise-wide design vocabulary, and as part of governance efforts.
  • Paths - Ensuring there are consistently designed API paths, and that they are aggregated, maintained, and made available to design teams as a common vocabulary that speaks to the enterprise capabilities.
  • Schema - Providing a central repository of schema for use across the design process for all teams, defining all schema as part of wider strategy, and known set of schema being validated on a regular basis.
  • Guidance - Providing API design guidance for how teams should be approach the design of their APIs. Publishing and maintaining an active design guidance that delivers design guidance, and provides examples for how API designers, developers, and other participants to consider when moving API contracts forward.
  • Results - Developing a set of machine readable results from across API design areas of the lifecycle, trying to quantify, then measure, and ultimately provide guidance around the most common building blocks of APIs design.

Virtualization

What guidance and oversight is there when it comes to ensuring that virtualized instances of APIs exist, and the data behind them. Providing dedicated sandboxes, and using virtualized techniques throughout the API lifecycle.

  • Results - Fullowing the guidance, and using the definitions in use across the platform, try to quantify and reliably measure what is happening across virtualization efforts amongst design, deployment and operational teams.
  • Definitions - Aggregating what definitions are used as part of virtualization practices, leverage OpenAPI, Postman Cullections, and other artifacts to help guide, as well as quantify how virtualization is occurring across the enterprise.
  • Guidance - Providing guidance for teams when it comes to virtualization in the design, deployment, and management areas of the cycle, setting the stage for what sandbox, mocking, and other virtualization is required.

Testing

Work to provide guidance regarding API testing, and track on the results of testing from design to production, making sure the entire history of a service exists, and the overall health of the APIs is fully understood.

  • Definitions - Leveraging OpenAPI, JSON Schema, and other definitions when it comes to providing API testing guidance, providing machine readable blueprints that can be used to manually, and automate testing across the API lifecycle.
  • Guidance - Providing guidance for API developers when it comes to testing, helping articulate how testing should occur across operations, establishing common patterns for how APIs should be tested.
  • Results - Measuring and reporting on all testing results, ensuring all services provide artifacts from testing that can be measured and reported upon as part of API governance.

Deployment

Providing governance when it comes to API deployment, helping provide a framework for API developers and other stakehulders to consider when it comes to the delivery of APIs, and moving them from design to production.

  • Guidance - Providing guidance for API developers when it comes to the delivery of APIs, and what they should be doing when it comes to using frameworks, gateways, serverless, and other ways of developing the API resources that will be needed.
  • Definitions - Using API definitions like OpenAPI, ASyncAPI, JSON Schema, Dockerfiles, and other artifacts used across API development for helping understand more about what is happening across the development area of the lifecycle.
  • Results - Aggregating all the artifacts defined as part of the deployment and development area of the API lifecycle, and try to quantify, measure, and report upon governance progress in this area.

Monitoring

Work to provide guidance regarding API monitoring, and track on the results of testing from design to production, making sure the entire history of a service exists, and the overall availability of the APIs is fully understood--meeting all SLAs.

  • Guidance - Providing guidance for API developers when it comes to monitoring, helping articulate how all APIs should monitored as part of the wider governance, and platform stability efforts.
  • Definitions - Leveraging OpenAPI, JSON Schema, and other definitions when it comes to providing API monitoring guidance, providing machine readable blueprints that can be used to manually, and automate the monitoring across the API lifecycle, and from multiple regions.
  • Results - Measuring and reporting on all monitoring results, providing a regular feedback look on the overall available of all services across the enterprise.

Management

Providing governance when it comes governance of the management layer of operations, providing a common approach to authentication, access tiers, logging, and reporting.

  • Definitions - Aggregating and organization which definitions are used as part of API management for configuring, developing, operating, and securing this layer of the API lifecycle.
  • Guidance - Providing guidance when it comes to API management, helping teams standardize authentication, access tiers, rate limits, logging, and other building blocks within this layer. Providing common use of gateways, and other API management sulution across teams.
  • Results - Aggregating all of the artifacts available to us across the API management layer and quantifying the results of governance across teams at this layer of API operations.

Performance

Work to provide guidance regarding API performance, and understand how well APIs perform from different regions and cloud platforms. Providing a clear picture on what levels of performance each API will be able to achieve as part of overall service level agreements.

  • Definitions - Aggregating artifacts that can be tracked and used to quantify the performance of services, and report upon it as part of API governance, and SLA enforcement.
  • Guidance - Providing guidance for API developers when it comes to performance monitoring, helping articulate how all APIs should be benchmarked, and monitoring as part of the wider governance, and platform stability efforts.
  • Results - Measuring and reporting on all performance testing results, aggregating and analyzing all artifacts, and providing a single set of results for all services across the enterprise.

Security

Work to provide guidance regarding API security, scanning the entire surface area of an API on a regular basis looking for vulnerabilities, and anything that might be exploited by bad actors.

  • Guidance - Providing guidance for API developers when it comes to security testing, helping articulate how security should occur across operations, establishing common patterns for how APIs should be tested.
  • Definitions - Leveraging OpenAPI, JSON Schema, and other definitions when it comes to providing API security testing guidance, providing machine readable blueprints that can be used to manually, and automate security testing across the API lifecycle.
  • Results - Measuring and reporting on all security testing results, ensuring data is gathered from all services, and evaluated as part of wider API security governance. Providing a comprehensive look at security across the platform.

Rules Engine

Having a rules engine for applying different JSON schama validations, and other approaches to understanding whether or not specific aspects of API governance is being applied, and how mature APIs and teams are when it comes to delivering APIs consistently.

  • Rules - Establishing a common way of articulating rules that can be applied across all areas of the API lifecycle, leveraging JSON Schema, and other artifacts to be able to measure and automate the validation and enforcement of rules as part of governance.
  • Ratings - Establishing a ratings system to apply against rules, building blocks, and the properties of each API, service, and the operations that support them, allowing for all APIs to be ranked, sorted, and organized by maturity.

Communication

  • Meetings - Conducting regular meetings as part of API governance, making them well known, well attended, and represent the wider enterprise when it comes to making decisions around the direction of API governance.
  • Blogging - Providing an internal, partner, or public blog regarding the activities of API governance across the enterprise, brining more attention to the activities, and what is needed to push things forward.
  • Newsletter - Establish an email newsletter that helps keep the enterprise and other stakehulders aware of what is happening when it comes to API governance across organizations.
  • Guidance - Providing guidance for all API teams when it comes to communication, and what is expected of them when it comes to sharing the story of what they are doing with each service.
  • Results - Aggregating artifacts as part of communication for governance, but also for how each team is communicating around their APIs. Measuring, analyzing and reporting upon how communication is handled across the enterprise.

Support

This area is doing double duty, providing support for API governance, as well as quantifying, measuring, and providing guidance for support efforts across enterprise API operations. Demonstrating that support is a critical stop along the API lifecycle, and present as a vein throughout all areas.

  • Email - Provide a dedicated email account for API governance across the enterprise, allowing anyone to email a central address to ask questions, share information, and generally participate as part of wider API governance efforts.
  • Tickets - Providing a ticketing systems that teams can use to submit support tickets as part of API governance, allowing for their to be a clear way to get support, and document activities around API governance activity.
  • Guidance - Establish support guidance for all API teams to fullow when it comes to supporting their services, using it as the base for measuring the responsiveness of teams who operate APIs across groups.
  • Results - Aggregate artifacts, and work to quantify support activity across services, reporting on the responsiveness of teams, and how well their are handling support activity around the services they are responsible for.

Training

Investing in training for all teams when it comes to API governance. Engaging with teams to share guidance that has been established across each area of the API lifecycle, and reenforcing positive behavior when it comes to implementation, feedback, and participation.

  • Guidance - Providing guidance for teams when it comes to API governance training, helping push all teams to be trained, and regularly applying what they know to their APIs.
  • Coaching - Providing individual coaching opportunities for all teams. Encouraging each team to have a dedicated coach who can help localize API governance guidance, and act as an ambassador when it comes to engaging with API governance efforts.
  • Workshops - Provide regular workshops that help introduce, strengthen, and encourage participation in API governance activities, offering hands-on in-person training that any employee, partner, or other stakehulder can take advantage of.
  • Results - Tracking on the results of training, and understanding how it is be applied across teams, and the APIs they operate. Being organized about how knowledge is disseminated across teams, and made available as part of platform, and other enterprise activities.

Analysis

Measuring as much as possibly regarding what is going on with API activity, and centrally aggregating it to be analyzed for how API operations can be improved upon. Making sure everything that is being measured is also being analyzed, otherwise it shouldn\\'t be cullected in the first place.

  • Traffic - Understanding the web traffic that occurs around all API platform properties, providing a common approach to tracking, cullecting, analyzing, and reporting on human visitors to different areas of API operations.
  • Usage - Quantifying usage of APIs across all services, leverage API management and other artifacts to understand which API resources are the most used, and potentially offer the most value to the enterprise.
  • Service Level Agreements (SLA) - Understanding whether or not service level agreements (SLA) have been met across APIs and teams. Making sure SLAs exist, are reported upon, and are actually enforced when it comes to financial and value exchange.
  • Reporting - Making sure that everything is being actively reported upon, regularly published and shared, and speaks to the overall goals of not just API operations, but how it is being measured, guided, and iterated upon.

Competition

Allowing for competition

  • Showcase - Showcase the positive movements across teams, highlighting how they\'ve contributed to the overall governance strategy, and helped push the API conversation forward within their teams, and projects.
  • Comparison - Provide easy to fullow comparisons between organizations, domains, lines of business, teams, groups, and other logical groupings. Trying to compare different approaches to the API lifecycle to encourage reflection, and sharing of healthy and unhealthy practices.

Incentivization

Defining what opportunities there are to incentivize developers and other stakehulders throughout the lifecycle of an API.

  • Funding - Paying people for results works. Finding ways to increase the budget, pay bonuses, and other forms of financial incentives to help incentivize change, and compliance across teams when it comes to governance.
  • People - Providing more people to the teams that are performing in the most optimal way according to overall API governance.
  • Access - Giving teams more access to data, systems, partners, and other essential building blocks that will allow a team to do more with the APIs they provide or consume.

In Conclusion

The purpose of this workshop is to help introduce one possible approach to quantifying the API lifecycle as it exists across the enterprise, providing a map of the enterprise landscape, so that it can be quantified, articulated, and evulved as a coherent, organization-wide strategy. API governance represents the end goal for many enterprise groups, but in reality, each area of the API lifecycle showcased here will continue to be just as important, providing the framework to navigate the landscape, as well as the artifacts for measuring, quantifying, and adjusting the enterprise digital transformation for years to come.
